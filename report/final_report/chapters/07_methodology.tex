\chapter{Experimental Methodology}
\label{chap:methodology}

\section{Research Questions}

The final evaluation is organised around five research questions.

\begin{itemize}
\item RQ1: How much does backend choice affect checkpoint loading time for real LLM checkpoints?
\item RQ2: How does checkpoint layout change the relative ordering of backends?
\item RQ3: How much overhead is introduced by the Python binding layer, and how much of that overhead can be reduced with binding-specific optimisation?
\item RQ4: To what extent do backend gains survive integration into a serving stack such as vLLM?
\item RQ5: What engineering lessons emerge when turning backend-specific microbenchmarks into a practical adaptive loading system?
\end{itemize}

\section{Experimental Platform}

All final experiments were run on a dedicated Linux server equipped with an NVIDIA H100 GPU and approximately 180~GB of host RAM. The machine was used as an isolated experiment host rather than as a shared development workstation. The software stack used for the final runs was Ubuntu~24.04, Python~3.12, and a current stable Rust toolchain. The experiments therefore target a realistic modern serving environment rather than the earlier local development machine used during prototype work.

Two details are important for interpreting the final results. First, the GPU-backed vLLM runs are constrained not only by raw device memory but also by vLLM's internal memory budgeting for KV cache, CUDA graphs, and runtime buffers. Second, the I/O experiments were conducted on Linux only, because the low-level \texttt{io\_uring} backend is Linux-specific.

\section{Backends Under Test}

The final system exposes four eager-loading policies at the Rust layer:

\begin{itemize}
\item \textbf{sync}: blocking POSIX reads with dynamic chunking and parallelism.
\item \textbf{async}: Tokio-based async loading with dynamic batching and per-file grouping.
\item \textbf{io\_uring}: a low-level Linux \texttt{io-uring} backend using multi-worker submission, dynamic ring sizing, and dynamic chunk planning.
\item \textbf{default}: an adaptive selector that chooses between the explicit backends based on workload structure.
\end{itemize}

The key methodological point is that the final \texttt{io\_uring} design is \emph{not} the original single-ring prototype. Early measurements showed that a single-ring, single-issuer design badly underutilised the machine and produced misleadingly poor results. The final backend uses multiple worker-owned rings, which materially changed the performance ranking. This architectural change is central to the interpretation of the results in Chapter~\ref{chap:results}.

The memory-mapped \texttt{open\_*} APIs are evaluated in the Python layer where they are practically useful, but they are not treated as eager-loading backends in the main Rust cold-load comparison.

\section{Checkpoint Formats and Model Ladder}

Two checkpoint layouts were evaluated.

\begin{itemize}
\item \textbf{SafeTensors}: whole-file or multi-shard eager loading.
\item \textbf{ServerlessLLM}: partitioned layout accessed through range-oriented loading.
\end{itemize}

The final model ladder consists of six open-access checkpoints:

\begin{enumerate}
\item \texttt{Qwen/Qwen3-0.6B}
\item \texttt{HuggingFaceTB/SmolLM3-3B}
\item \texttt{Qwen/Qwen3-4B}
\item \texttt{Qwen/Qwen3-8B}
\item \texttt{Qwen/Qwen3-14B}
\item \texttt{Qwen/Qwen3-32B}
\end{enumerate}

This ladder was chosen to cover small, medium, and large checkpoints while also including one cross-family model (SmolLM3-3B) to reduce the risk of overfitting conclusions to a single vendor family.

\section{Cache-Control Procedure}

Cold-cache behaviour is the primary condition for the Rust and Python eager-loading measurements. The final experiments do \emph{not} use in-code cache eviction. Earlier prototype code used \texttt{posix\_fadvise} and related page-cache manipulation, but that approach was removed from the final system.

Instead, cold-cache runs use a manual Linux-side procedure before each benchmark invocation:

\begin{enumerate}
\item run \texttt{sync}
\item run \texttt{echo 3 > /proc/sys/vm/drop\_caches}
\item invoke the benchmark binary or Python benchmark
\end{enumerate}

Warm-cache conditions skip the cache-drop step and reuse the existing page-cache state. This final procedure matches the actual scripts and experiment harness used to generate the saved result artifacts.

\section{Rust-Layer Benchmark Design}

Rust-layer measurements were generated with the standalone profiling harness. Two categories of Rust experiments were used.

\subsection{Anchor Repetitions}

A five-repetition cold-cache anchor set was collected for:

\begin{itemize}
\item \texttt{Qwen/Qwen3-0.6B}
\item \texttt{Qwen/Qwen3-8B}
\item \texttt{Qwen/Qwen3-32B}
\end{itemize}

for both \texttt{SafeTensors} and \texttt{ServerlessLLM}, across \texttt{sync}, \texttt{async}, \texttt{io-uring}, and \texttt{default}. These repeated measurements were used to identify stable workload regimes and to redesign the \texttt{default} selector.

\subsection{Full Cold Matrix}

A full cold-cache matrix was then run across the six-model ladder, both formats, and the four eager backend policies. The resulting artifact is stored under the H100 profile results directory as \texttt{full\_cold\_matrix.tsv}. This matrix is the authoritative source for backend-attribution claims in this report.

The Rust layer is treated as the backend-attribution layer because it isolates I/O and checkpoint processing without Python tensor materialisation overhead or serving-engine initialisation overhead.

\section{Python Benchmark Design}

Python benchmarks were run against the public binding surface rather than against every internal Rust backend directly. This reflects the actual intended Python API:

\begin{itemize}
\item \texttt{load\_*}: adaptive default policy
\item \texttt{load\_*\_sync}: explicit synchronous eager load
\item \texttt{load\_*\_async}: explicit Tokio eager load
\item \texttt{open\_*}: mmap-backed lazy access path
\end{itemize}

The main Python benchmark model was \texttt{Qwen/Qwen3-8B}. This model is large enough to make backend differences visible, while still being practical for repeated user-facing benchmark runs.

The Python layer was benchmarked before and after two types of improvement:

\begin{enumerate}
\item binding/runtime overhead reduction (shared Tokio runtime, reduced repeated Torch conversion work)
\item selector redesign in the Rust core
\end{enumerate}

This allowed the evaluation to separate pure backend effects from binding-layer engineering effects.

\section{vLLM Benchmark Design}

vLLM benchmarks were run through a custom benchmark loader integration. Unlike the Python eager-loading benchmarks, the vLLM evaluation measures the loading backends inside a wider serving-engine context.

Three benchmark kinds were collected:

\begin{itemize}
\item \textbf{load\_only}: model initialisation time only
\item \textbf{TTFT}: time to first token, including initialisation and first generation step
\item \textbf{steady\_state\_decode}: average per-request decode latency after warm-up
\end{itemize}

Both warm and cold cache conditions were collected. The full vLLM matrix covers all six fixtures and all available benchmark loaders. After initial failures on the 14B and 32B models, the vLLM harness was rerun with a higher \texttt{gpu\_memory\_utilization} and a reduced \texttt{max\_model\_len}. These reruns resolved the large-model failures, yielding a final matrix with no failed rows.

\section{Result Artifacts and Reproducibility}

All final experiment artifacts are stored locally under \texttt{results/h100/} and are organised into four subdirectories:

\begin{itemize}
\item \texttt{results/h100/analysis}: profiling notes and bottleneck analysis
\item \texttt{results/h100/profile}: Rust-layer benchmark outputs
\item \texttt{results/h100/python}: Python benchmark JSON artifacts
\item \texttt{results/h100/vllm}: vLLM summary TSVs and raw logs
\end{itemize}

The scripts retained in \texttt{scripts/} are the small final set actually used for the completed experiment flow: anchor repetitions, full Rust cold matrix, full vLLM matrix, and failed-row reruns.

\section{Fairness and Interpretation Rules}

Three rules were used to keep the evaluation interpretable.

\begin{enumerate}
\item Rust-layer results are used for backend attribution, because they isolate checkpoint loading from Python and serving-engine overhead.
\item Python-layer results are used to quantify binding overhead and to validate whether the public \texttt{default} API tracks the stronger explicit backend in practice.
\item vLLM results are treated as end-to-end serving initialisation evidence, not as a pure backend microbenchmark.
\end{enumerate}

This separation is important because backend gains do not propagate unchanged across layers. A backend that wins clearly in the Rust harness may yield a smaller gain in Python, and a smaller gain still once vLLM initialisation dominates the critical path.

\section{Threats to Validity}

The main threats to validity are as follows.

\textbf{Hardware specificity.} Absolute times are tied to the H100-based Linux environment and its storage stack. The results should therefore be interpreted primarily as relative backend comparisons under this configuration.

\textbf{Selector dependence.} The \texttt{default} policy changed during the project as evidence accumulated. This is intentional: the report evaluates the final selector, but also discusses how earlier architectural and heuristic choices produced different outcomes.

\textbf{Serving-stack configuration.} vLLM feasibility for the largest models depended on harness configuration, especially KV-cache budgeting. The report therefore distinguishes backend limitations from serving-engine configuration limitations.

\textbf{Layered overheads.} Python conversion and serving-engine initialisation can obscure pure I/O effects. This is why the report keeps Rust, Python, and vLLM layers analytically separate rather than collapsing them into a single headline metric.
