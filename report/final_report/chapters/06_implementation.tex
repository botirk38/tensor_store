\chapter{Implementation and Software Engineering Process}
\label{chap:implementation}

\section{Rust Core Implementation}

The Rust core of TensorStore is implemented in the main crate and provides four kinds of functionality: format parsing, backend execution, conversion between formats, and command-line tooling for profiling and conversion. The implementation is organised so that format logic, backend logic, and benchmark logic remain distinct.

The format layer contains the \texttt{SafeTensors} and \texttt{ServerlessLLM} implementations. The backend layer contains the synchronous, async, io\_uring, and mmap-backed loading paths. The converter layer provides SafeTensors-to-ServerlessLLM conversion. The CLI binaries provide small operational tools for conversion, profiling, and demonstration.

\section{Backend Implementation}

The final backend implementation reflects the main engineering lessons of the project.

\subsection{Synchronous Backend}

The synchronous backend uses blocking file I/O but does not stay purely serial. Large whole-file loads are chunked and executed with host-side parallelism. Range-batch requests use coalescing and dynamic concurrency. This design is intentionally stronger than a naive one-thread baseline, because the project aims to compare practical backend implementations rather than caricatures.

\subsection{Async Backend}

The async backend is implemented on top of Tokio. The final version groups work by file, limits unnecessary task fanout, and uses coalesced range-batch execution where appropriate. This backend proved particularly relevant for range-heavy \texttt{ServerlessLLM} workloads, where grouped per-file execution is often a good fit for the request structure.

\subsection{io\_uring Backend}

The final \texttt{io\_uring} backend uses the low-level \texttt{io-uring} crate~\cite{io_uring_crate} rather than \texttt{tokio-uring}. The most important implementation change was architectural: the early prototype used one shared ring, but the final backend uses multiple worker-owned rings with lazy ring initialisation, dynamic ring sizing, dynamic chunk planning, and dynamic in-flight depth. This redesign was driven by profiling evidence rather than by a priori preference.

This matters because the final results depend on the final architecture, not just on the backend family label. A poorly designed \texttt{io\_uring} backend underperformed badly; a better designed one became competitive or best in several important regimes.

\subsection{Buffer Management}

The implementation uses pooled buffers for the eager-loading backends, plus aligned buffers where direct I/O makes sense. Buffer pooling reduces repeated allocation pressure in hot loading paths. This design was preserved deliberately even as other parts of the system were cleaned up, because it reflects a real implementation concern rather than a benchmark-only optimisation.

\section{Format Implementations}

The \texttt{SafeTensors} model loader discovers shard files, parses the header metadata, and loads tensors through the selected backend. The final selector does not use an opaque synthetic cost model; instead, it uses a small workload-aware regime decision driven primarily by shard count and total bytes.

The \texttt{ServerlessLLM} model loader reads the partition index, groups and coalesces range requests, and then loads the required data through the selected backend. The selector is again workload-aware, but the relevant signals differ because the partitioned format produces a range-heavy workload.

The converter from \texttt{SafeTensors} to \texttt{ServerlessLLM} was also an important implementation focus. Earlier converter iterations reloaded whole files repeatedly and did not always respect planned destination offsets. The final converter writes partition files correctly and is covered by roundtrip and failure-path tests.

\section{Python Bindings}

The Python bindings are implemented in \texttt{bindings/python/rust\_ext/src/}. Public Rust functions are exposed through PyO3 wrappers.

For synchronous loading, the Rust function is called directly and the GIL is released during I/O using \texttt{py.allow\_threads()}. For mmap-backed lazy access, Rust handle types are wrapped as Python classes. For async loading, the bindings run the Rust async path on a shared Tokio runtime. This avoids repeatedly constructing a runtime for each call and was one of the practical optimisations that improved Python benchmark results.

The binding layer also required optimisation of Torch conversion. The final implementation reuses a shared runtime, reduces repeated dtype lookups, and cuts down repeated conversion work when materialising tensors into Python dictionaries.

\section{vLLM Loader Integration}

The vLLM integration lives in the Python benchmark layer rather than in the public end-user API. This is intentional. The public Python API keeps \texttt{io\_uring} behind \texttt{default}, while the benchmark layer exposes explicit loader paths so that the serving experiments can isolate loader behaviour.

The vLLM loader path reuses vLLM's normal checkpoint-preparation logic and substitutes the TensorStore-backed loading step. In the final benchmark matrix, this loader family was tested across all six fixtures, warm and cold cache conditions, and all three benchmark kinds.

\section{Module Structure}

The final codebase is structured around a small number of high-value modules rather than a large number of benchmark-specific fragments.

\begin{itemize}
\item \texttt{src/backends/}: core eager-loading implementations and shared planning logic
\item \texttt{src/formats/safetensors/}: SafeTensors loaders and serializer
\item \texttt{src/formats/serverlessllm/}: ServerlessLLM loaders, index, serializer, and helpers
\item \texttt{src/converters/}: format conversion logic
\item \texttt{src/bin/convert/}: conversion CLI
\item \texttt{src/bin/profile/}: profiling CLI used for the Rust benchmark matrices
\item \texttt{bindings/python/rust\_ext/}: PyO3 binding layer
\item \texttt{bindings/python/benchmarks/}: Python and vLLM benchmark harnesses
\end{itemize}

This structure matters to the software-engineering story of the project. One of the later cleanup phases removed many one-off scripts and stale interfaces so that the final submission would reflect the actual completed experiment flow rather than every intermediate iteration.

\section{Build and Validation Process}

The Rust crate is built with Cargo, and the Python bindings are built with \texttt{maturin}. The final validation process used three layers of checking.

\begin{enumerate}
\item Rust correctness and linting: \texttt{cargo test}, \texttt{cargo clippy}, and Linux cross-build checks.
\item Python binding correctness: unit and integration tests for the public binding surface.
\item Benchmark-level validation: Rust matrices, Python benchmarks, and the completed vLLM matrix saved under \texttt{results/h100/}.
\end{enumerate}

This layered validation strategy was important because the project's contribution spans more than one execution layer. A fast backend alone is not enough if the binding layer or serving integration quietly nullifies the gain.

\section{Testing Strategy}

The final testing strategy combines unit, integration, and benchmark validation.

Rust unit tests cover format parsing, batching/coalescing logic, selector boundaries, backend parity, converter error cases, and converter roundtrips. Linux-specific correctness tests were added for the \texttt{io\_uring} backend, and additional parity tests were added for the synchronous and async backends. CLI smoke coverage is included inline in the binaries so that help surfaces and basic error cases are also tested.

Python unit tests cover the public binding API, including Linux-only \texttt{io\_uring} surfaces where appropriate. Additional tests were added for the vLLM benchmark loader dispatch logic to ensure that the serving benchmarks were not relying on untested glue code.

The completed benchmark artifacts provide a third layer of validation. The project therefore treats benchmarking not only as performance evaluation but also as a form of high-level system validation.

\section{Implementation Problems and Resolutions}

Two implementation problems were especially important.

The first was cold-cache measurement drift. Early runs were polluted by page-cache reuse, which made supposedly cold runs behave like warm runs. The final resolution was to remove in-code cache eviction and adopt an explicit Linux-side cold-cache procedure.

The second was the early single-ring \texttt{io\_uring} architecture. Profiling showed that the backend was underutilising the machine and spending too much time in one submission/completion path. The redesign to multi-worker rings changed both the performance and the interpretation of the entire project. It is therefore not just an implementation detail, but one of the central pieces of technical decision-making in the report.
