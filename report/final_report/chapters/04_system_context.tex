\section{The Cold-Start Pipeline in Serverless LLM Inference}

When a serverless inference platform receives a request for a model that is not currently resident in memory, it must reload the model's checkpoint from persistent storage before inference can begin. This sequence of steps constitutes the cold-start pipeline.

The pipeline begins when a request arrives for an absent model. The platform identifies the checkpoint location and initiates loading. The loader reads checkpoint metadata to determine the set of tensors and their storage locations. It then issues read requests to retrieve each tensor's data. As data arrives, it is copied into the target memory location. Once all tensors are loaded, the model initialises its runtime state. Inference then proceeds.

Checkpoint loading occupies the longest segment of this pipeline for large models. The duration depends on the total checkpoint size, the storage device bandwidth, the number and size of individual read requests, and the efficiency of the submission path between the loader and the storage device.

\section{Experimental Constraints}

This project evaluates checkpoint loading in isolation, separate from the broader inference pipeline. This isolation is deliberate. By holding the checkpoint layout and measurement path constant across backends, differences in loading performance can be attributed more cleanly to the loading mechanism rather than to model initialisation, runtime setup, or scheduling effects.

One design decision requires explicit explanation: the project begins from a fixed workload view so that the storage-facing execution path can be compared fairly. This constraint is not a claim that real loaders cannot merge, split, or reorder byte-range requests. Rather, it is a methodological choice that prevents application-level scheduling changes from overwhelming backend-level comparisons during the core Rust experiments.

In practice, checkpoint format designers and loader authors can make different trade-offs. The implications of different application-level scheduling policies are beyond the scope of this evaluation.

\section{The Submission Bottleneck Hypothesis}

The central hypothesis of this project is that checkpoint loading performance is often limited by the I/O submission path rather than by storage bandwidth. However, the final report refines this hypothesis in an important way: whether a more complex backend helps depends on workload shape. Submission depth, queueing, and batching matter, but they only translate into gains when the checkpoint layout and model size create enough useful in-flight work to justify them.

Under io\_uring, submission and completion are decoupled. An application can prepare many requests in advance and submit them in batches. The kernel processes them asynchronously and delivers completion events separately. This model can maintain a higher useful queue depth than a simpler blocking path, but the final project shows that the benefit appears only once the loader architecture and the workload regime are matched properly.

The refined hypothesis therefore predicts not that io\_uring always wins, but that checkpoint loading should be driven by adaptive heuristics. In the clearest high-level form, smaller eager loads should favour the simpler synchronous path, while larger checkpoints should increasingly reward the deeper submission strategy provided by io\_uring. Whether this heuristic survives Python bindings and a full serving pipeline is the empirical question the rest of the report addresses.
